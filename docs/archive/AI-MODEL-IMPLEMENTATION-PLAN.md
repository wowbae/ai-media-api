# План внедрения отдельного режима `/ai-model`

## 1) Цель

Сделать отдельный UI-режим для генерации контента под AI-модель на маршруте `/ai-model` без стандартной авторизации, с визуальным отличием (изумрудный акцент), отдельной библиотекой изображений и расширенным prompt-flow:

- обычный `prompt`
- `enhancedPrompt` (улучшенный)
- `negativePrompt`
- кнопка `✨` для авто-улучшения через Grok

Поддерживаемые модели в этом режиме:

- `Nano Banana 2 BP (Kie.ai)`
- `Z-Image (Wavespeed)`

Дополнительно для `Z-Image (Wavespeed)`:

- поддержать отправку LoRA (помимо референсных изображений), если выбран именно этот model preset.

## 2) Важное архитектурное решение: изоляция режимов

Нужно ввести явный признак режима генерации во всех сущностях, где храним и читаем медиа:

- `appMode: "default" | "ai-model"`

Это должно применяться к:

- запросам генерации
- файлам/медиа в библиотеке
- history/list endpoints
- фильтрации на фронте

Без этого разделение библиотек будет ненадежным.

## 3) Почему “роут без авторизации” рискованный подход

Скрытый URL сам по себе — слабая защита (security through obscurity). Минимально безопасный вариант при сохранении UX “без логина”:

- проверка `x-ai-model-key` (секрет в env) только для `/ai-model` API
- либо allowlist по IP (если у тебя фиксированный IP)
- отключить индексирование (`noindex`) и не показывать ссылку в обычной навигации

Если хочешь строго “вообще без проверки” — можно, но это повышает риск утечки и несанкционированного использования провайдеров.

## 4) Backend: этапы внедрения

## Этап A. Добавить режим в контракты и БД

1. Расширить типы/DTO:
    - `GenerateParams` и связанные request-схемы
    - добавить `appMode` с дефолтом `"default"`
2. Миграция БД:
    - добавить поле `app_mode` в таблицы генераций/медиа (минимум там, где строится библиотека)
    - индекс по `app_mode` + дате (`created_at`) для быстрых выборок
3. Обновить сохранение результатов:
    - при `saveFilesToDatabase` писать `appMode` из исходного запроса

## Этап B. Разделить библиотеку медиа

1. Во всех endpoints чтения библиотеки (`/files`, `/requests`, связанные list/chats endpoints) добавить фильтр по `appMode`
2. Никогда не возвращать смешанные данные между `"default"` и `"ai-model"`
3. На сервере сделать дефолт `"default"`, чтобы старый UI работал без изменений

## Этап C. Ограничить модели для `/ai-model`

1. В генерации добавить валидацию по режиму:
    - `default`: текущий whitelist
    - `ai-model`: только `Nano Banana 2 BP (Kie.ai)` и `Z-Image (Wavespeed)`
2. Если модель не разрешена для режима — `400` с понятным текстом
3. Для `Z-Image (Wavespeed)` зафиксировать контракт LoRA:
    - `loras[]` optional
    - максимум 3 элемента
    - каждый элемент: `path: string`, `scale?: number`
    - `path` может быть ID LoRA или URL `.safetensors` (по документации Wavespeed)

## Этап D. API для улучшения промпта через Grok

Добавить endpoint, например `POST /media/prompt-enhance`:

Вход:

- `prompt: string`
- `attachments: string[]` (URL/base64 по текущему формату)
- `appMode: "ai-model"` (валидация)

Выход:

- `enhancedPrompt: string`
- `negativePrompt: string`

Логика:

1. Собрать мультимодальный запрос в Grok
2. Использовать модель `grok-4.20-0309-reasoning`
3. Запрашивать строго структурированный JSON-ответ
4. Валидировать ответ (обязательные поля, fallback при частичном ответе)
5. Таймаут + retries + аккуратный error message на фронт

Примечание: модель и параметры вынести в shared-константы (по текущим правилам проекта).

## 5) Frontend: этапы внедрения

## Этап E. Новый маршрут и визуальная идентификация

1. Добавить route `/ai-model` (TanStack Router)
2. Переиспользовать текущий экран генерации, но в режиме `appMode="ai-model"`
3. Поменять акцентный цвет на изумрудный через theme token/variant
4. Не показывать этот роут в основном меню

## Этап F. Поля prompt и кнопка `✨`

В chat-input добавить:

- основное поле `prompt` (как сейчас)
- поле `enhancedPrompt` (read/write)
- поле `negativePrompt` (read/write)
- кнопку `✨` рядом с кнопкой генерации

Поведение `✨`:

1. Берем текущий `prompt` + прикрепленные изображения
2. Вызываем `POST /media/prompt-enhance`
3. Подставляем:
    - `enhancedPrompt` в поле “улучшенный промпт”
    - `negativePrompt` в соответствующее поле
4. Ошибки показываем как non-blocking toast/alert

Генерация в `/ai-model`:

- отправляет `appMode: "ai-model"`
- в payload включает `enhancedPrompt` и `negativePrompt`
- если выбрана модель `Z-Image`, дополнительно отправляет `loras[]` (если заполнены)

## Этап F2. LoRA UX для `/ai-model` (без дублирования интерфейса)

В проекте уже есть рабочие поля LoRA и backend-валидация. Для `/ai-model`:

1. Не добавлять второй/параллельный интерфейс LoRA, а переиспользовать текущий.
2. Показывать LoRA-поля только при выбранной модели `Z-Image`.
3. Для `Nano Banana 2 BP` скрывать LoRA-UI, чтобы не путать режимы.
4. Серверная валидация остается источником истины, фронт — только UX-подсказки.
5. Явно указать в UI, что LoRA передается отдельно от референсных изображений.

## Этап G. Отдельная библиотека на фронте

1. Все запросы библиотеки из `/ai-model` отправляют `appMode: "ai-model"`
2. Обычный роут продолжает использовать `appMode: "default"`
3. UI-лейбл режима (например badge `AI MODEL`) для исключения путаницы

## 6) SSE и обновления статусов

Учитывая правила проекта (SSE, не polling на клиенте):

- SSE payload должен содержать `appMode`
- клиент обрабатывает только события своего режима
- иначе возможны “чужие” обновления в UI

## 8) Критерии готовности (Definition of Done)

1. `/ai-model` работает отдельно и визуально отличается (изумрудный акцент)
2. Библиотеки медиа между `default` и `ai-model` полностью изолированы
3. В `/ai-model` доступны только 2 нужные модели
4. Кнопка `✨` заполняет `enhancedPrompt` и `negativePrompt` через Grok
5. Генерация учитывает новые поля и проходит end-to-end
6. SSE-события не смешивают режимы
7. Проверены негативные сценарии (ошибка Grok, пустые вложения, таймауты)
8. Для `Z-Image` корректно работает LoRA-сценарий: 0 LoRA, 1 LoRA, 3 LoRA; при 4+ — ожидаемая ошибка валидации

## 9) Порядок реализации (рекомендуемый)

1. Ввести `appMode` в типы + БД + backend фильтрацию
2. Добавить `/ai-model` route и изумрудную тему
3. Ограничить модели по режиму
4. Сделать `prompt-enhance` endpoint и интеграцию `✨`
5. Прогнать e2e-проверку обоих режимов
6. Прогнать сценарии LoRA для `Z-Image` в `/ai-model`

## 10) Открытые вопросы перед реализацией

1. Нужна ли минимальная защита `/ai-model` через секретный ключ в header (рекомендую: да)?
2. Какой из prompt идет в фактическую генерацию по кнопке “Generate”:
    - только `enhancedPrompt`, если заполнен
    - или всегда исходный `prompt` + `enhancedPrompt` как доп-контекст?
3. Нужно ли сохранять в БД оба поля (`enhancedPrompt`, `negativePrompt`) для истории и повторной генерации?
4. Нужен ли отдельный preset-список LoRA (select) поверх текущего ручного ввода `path`, или пока оставляем только ручной ввод?

## 11) Примечание по документации Wavespeed (проверено)

По странице модели `wavespeed-ai/z-image/turbo-lora`:

- LoRA поддерживается нативно;
- можно передавать до 3 LoRA в одном запросе;
- формат LoRA: `path` + optional `scale`;
- помимо LoRA используются стандартные поля (`prompt`, `size`, `seed`).

Это согласуется с уже реализованным контрактом в текущем backend/frontend.
